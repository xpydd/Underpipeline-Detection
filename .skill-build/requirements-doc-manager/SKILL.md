# 需求文档管理

## 概览

使用这个 skill 生成或维护中文 Markdown 需求文档，并在每次迭代时同步更新版本号、日期和正文内容。
默认目标文件必须放在项目根目录，文件名固定为 `product-spec.md`，不要放进 `docs/` 或其他文件夹，除非用户明确指定了不同路径。

## 工作流

### 1. 确认文档目标与输出路径

优先确认以下信息：

- 文档服务的项目或模块名称
- 目标文件路径
- 当前迭代或版本阶段
- 本轮新增、修改、删除的需求点
- 需求来源：会议纪要、聊天记录、原型、代码、工单、任务拆解或历史文档

如果用户没有给出路径，默认使用项目根目录下的：

```text
product-spec.md
```

### 2. 初始化文档框架

当目标文件不存在时，先运行：

```bash
python scripts/upsert_requirement_doc.py --project-name "项目名称"
```

脚本会创建中文 Markdown 文档骨架，并写入标题、项目名称、文档版本、更新日期、所属迭代和文档状态。
不要手动删除脚本插入的元数据注释块，后续自动更新时间和版本号会依赖它。
如果未显式传入 `--path`，脚本必须默认在当前项目根目录生成 `product-spec.md`。

### 3. 建立需求上下文

更新正文前，先阅读并吸收以下信息：

- 当前需求文档
- 历史需求变更日志
- 与本轮迭代有关的原型、页面、代码和数据结构
- 历史讨论中已经确认的业务规则
- 待确认项和仍有争议的范围界定

如果信息不足，优先在文档中显式标记“待确认事项”，不要伪造结论。

### 4. 更新文档头部元数据

每次需求发生有效变化时，都要同步更新：

- 文档版本
- 更新日期
- 所属迭代
- 文档状态

版本号统一采用 `v主版本.次版本.修订号`，并遵循：

- 大版本：需求范围重构、目标变化、核心流程改版、正式基线切换
- 次版本：新增功能、明显扩展业务规则、补充完整模块需求
- 修订号：措辞澄清、错别字修正、结构整理、对既有结论的轻量补充

如果用户没有明确指定版本号，先根据改动幅度做合理判断，并在文档或变更日志中写明判断依据。

### 5. 更新正文内容

正文必须保持中文，并优先覆盖以下结构：

- 需求背景
- 目标与范围
- 用户角色与业务场景
- 功能需求
- 业务规则
- 非功能需求
- 风险与依赖
- 验收标准
- 本次迭代更新摘要
- 待确认事项

只在确有必要时增删章节；不要为了“完整”而保留空洞内容。对于暂未确定的信息，写出当前假设、影响和待确认动作。

### 6. 与需求变更日志保持同步

当需求文档被创建或发生内容变化时，同步维护需求变更日志，默认路径为：

```text
spec-log.md
```

如果日志文件不存在，创建它；如果已经存在，追加本轮记录。记录至少包含：

- 日期
- 版本号
- 变更类型
- 变更摘要
- 影响范围
- 关联迭代
- 状态

需要时配合 `$requirements-change-log-manager` 一起使用，但即使单独触发本 skill，也要主动维护日志文件。

## 更新策略

### 新建文档时

- 先生成骨架
- 再把现有信息整理进对应章节
- 明确标出缺失信息和确认边界
- 同步写入第一条“初始化”变更记录

### 迭代更新时

- 先阅读原文，避免覆盖已确认内容
- 把本轮变化映射到具体章节，而不是只在文末追加说明
- 用“本次迭代更新摘要”概括本轮差异
- 同步提升版本号与更新日期
- 同步更新需求变更日志

### 重构文档结构时

- 保留既有结论，不要因为重排结构而丢失信息
- 如果旧章节被拆分或合并，在变更日志中说明原因
- 仅在结构确实阻碍可读性时才大改

## 质量要求

- 全文使用中文
- 日期统一使用 `YYYY-MM-DD`
- 版本号统一使用 `v0.1.0` 这种格式
- 标题与章节命名保持稳定，减少无意义改名
- 验收标准要可验证，避免只写“支持”“优化”“完善”
- 业务规则要具体到条件、动作、限制、异常场景
- 待确认事项要写清楚“缺什么”“影响什么”“谁来确认”

## 推荐命令

初始化或更新文档头部：

```bash
python scripts/upsert_requirement_doc.py --project-name "项目名称" --version "v0.2.0" --date "2026-04-09" --iteration "迭代二"
```

如果用户没有给出版本号或日期，可以省略；脚本会为新文档填入默认值，并为现有文档补齐或覆盖元数据。
如果用户要求自定义路径，再显式传入 `--path`；否则始终优先使用项目根目录 `product-spec.md`。

## 参考资料

- 模板与编写规范：阅读 [references/requirements-doc-template.md](references/requirements-doc-template.md)
- 版本与同步原则：优先遵守本 skill 的版本规则；如果项目已有版本制度，则以项目制度为准
- 文件落点原则：默认文件必须位于项目根目录，文件名固定为 `product-spec.md`

## 完成前检查

- 需求文档是否存在且为中文 Markdown
- 版本号和日期是否已更新
- 本次变化是否已落到具体章节
- 待确认事项是否被显式标记
- 需求变更日志是否已同步更新
