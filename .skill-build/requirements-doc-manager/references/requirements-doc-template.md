# 中文需求文档模板与编写规范

默认输出文件位于项目根目录，文件名固定为 `product-spec.md`。
除非用户明确指定，不要创建 `docs/` 或其他子目录。

## 元数据规范

需求文档顶部保留以下元数据字段，并全部使用中文：

- 项目名称
- 文档版本
- 更新日期
- 所属迭代
- 文档状态

推荐格式：

```md
# 需求文档

<!-- 需求文档元数据：开始 -->
- 项目名称：示例项目
- 文档版本：v0.1.0
- 更新日期：2026-04-09
- 所属迭代：迭代一
- 文档状态：草稿
<!-- 需求文档元数据：结束 -->
```

## 推荐章节

```md
## 1. 需求背景
## 2. 目标与范围
## 3. 用户角色与业务场景
## 4. 功能需求
### 4.1 功能清单
### 4.2 详细需求
## 5. 业务规则
## 6. 非功能需求
## 7. 风险与依赖
## 8. 验收标准
## 9. 本次迭代更新摘要
## 10. 待确认事项
```

## 编写要求

- 需求背景说明为什么做
- 目标与范围说明做什么、不做什么
- 用户角色与业务场景说明谁在什么情境下使用
- 功能需求优先写清触发条件、输入、输出、限制和异常
- 业务规则优先写判断条件、状态流转、边界场景
- 验收标准必须可验证
- 待确认事项说明缺失信息及影响

## 版本规则

- 大版本：目标或主流程重构、基线升级
- 次版本：新增功能或明显扩展
- 修订号：措辞、结构、澄清与轻量修正

## 与变更日志的关系

只要需求文档版本、范围、规则或结构发生变化，就同步更新需求变更日志。
如果仅有排版级别的细小调整，也建议至少追加一条“澄清”或“修订”记录。
