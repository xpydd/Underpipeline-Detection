# 需求变更日志管理

## 概览

使用这个 skill 创建或维护中文需求变更日志，并确保每次需求迭代都有可追溯的版本记录。
默认目标文件必须放在项目根目录，文件名固定为 `spec-log.md`，不要放进 `docs/` 或其他文件夹，除非用户明确指定了不同路径。

## 工作流

### 1. 确认日志目标与来源

先确认这些输入：

- 对应的需求文档路径
- 当前或目标版本号
- 变更日期
- 变更类型：新增、修改、删除、重构、澄清、初始化等
- 变更原因与背景
- 影响范围：页面、流程、角色、接口、报表、权限、数据规则等
- 关联迭代和当前状态

如果用户没有给出路径，默认使用项目根目录下的：

```text
spec-log.md
```

### 2. 初始化日志文件

日志文件不存在时，先运行：

```bash
python scripts/upsert_requirement_change_log.py --project-name "项目名称"
```

脚本会创建中文 Markdown 日志文件，并写入元数据、表头和首条初始化记录。
如果未显式传入 `--path`，脚本必须默认在当前项目根目录生成 `spec-log.md`。

### 3. 每次迭代都追加记录

只要需求文档出现实质性变化，就追加一条记录。不要只更新时间而不写明变化内容。
每条记录至少包含：

- 日期
- 版本
- 变更类型
- 变更摘要
- 影响范围
- 关联迭代
- 状态

如果需要补录历史变更，也按时间倒序补齐。

### 4. 更新日志头部元数据

每次记录新变更后，更新日志头部中的：

- 当前版本
- 最近更新日期
- 维护状态

版本号规则与需求文档一致，统一采用 `v主版本.次版本.修订号`。

### 5. 与需求文档强绑定

日志不是独立文档，而是需求文档的历史视图。维护日志时必须校验对应需求文档是否同步：

- 需求文档版本是否一致
- 更新日期是否一致或合理
- 重大结构调整是否已有记录
- 删除或废弃项是否说明原因和影响

必要时配合 `$requirements-doc-manager` 一起使用；即使只触发本 skill，也要主动检查需求文档。

## 记录规范

### 变更类型建议

- 初始化：第一次建立需求文档或日志文件
- 新增：新增功能、流程、规则、页面、角色能力
- 修改：原有需求被调整、扩展、收缩或替换
- 删除：需求下线、取消、废弃
- 重构：结构重排、章节拆分、逻辑重新组织
- 澄清：对歧义、边界、规则、命名进行明确说明

### 变更摘要写法

变更摘要要回答“改了什么”。尽量使用一句话说明动作和对象，例如：

- 新增项目数据页的筛选与导出需求
- 调整业主管理列表的权限范围与查询条件
- 删除旧版报表入口并统一到知识中心

### 影响范围写法

影响范围要回答“影响到哪里”。优先写具体对象，例如：

- 项目数据页、导出接口、列表筛选逻辑
- 业主管理页面、角色权限、后端查询参数
- 报表导航、菜单权限、历史入口跳转

### 状态写法

状态建议使用：

- 已记录
- 待确认
- 已合并
- 已废弃

## 推荐命令

仅更新日志头部：

```bash
python scripts/upsert_requirement_change_log.py --project-name "项目名称" --version "v0.2.0" --date "2026-04-09"
```

更新头部并追加变更记录：

```bash
python scripts/upsert_requirement_change_log.py --project-name "项目名称" --version "v0.2.0" --date "2026-04-09" --append-entry --change-type "新增" --summary "新增项目数据页导出需求" --impact "项目数据页、导出逻辑、权限控制" --iteration "迭代二" --entry-status "已记录"
```

## 完成前检查

- 日志文件是否存在且为中文 Markdown
- 当前版本和最近更新日期是否已同步
- 本轮变更是否有单独记录
- 影响范围是否具体
- 对应需求文档是否已同步更新
